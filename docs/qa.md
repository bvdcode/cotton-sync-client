Cotton Sync Windows virtual files post-release plan

This plan tracks the Windows virtual files/placeholders epic after the first full-mirror release. A checkbox is complete only when the implementation exists, focused tests pass, and the listed evidence is recorded. `PARTIAL`, `BLOCKED`, `FAIL`, and `NOT IMPLEMENTED` do not count as complete.

Windows virtual files must not be counted as a passed first-release gate until every required implementation, automated, and manual checkbox here is implemented and verified.

### Verification Discipline

- Single checklist items must be verified with the fastest focused evidence that proves that item: targeted unit/integration test, CLI smoke, log sample, screenshot, DB query, or installed-artifact observation.
- Do not start a full solution build, full test suite, package workflow, release workflow, or installer rebuild for one isolated checkbox.
- Batch expensive verification after meaningful groups of source changes, packaging changes, or release-gate changes. One expensive run must close or validate multiple related checkboxes.
- If an expensive workflow is required, record its run id, target commit, reason, and expected follow-up, then continue with other fast checks instead of waiting inline.
- For manual desktop-sync workflow runs, use `package_target` to build only the artifact family needed for the current gate (`desktop-windows`, `desktop-linux`, `cli-windows`, or `all`).
- A checkbox can be marked `[x]` only for a clean pass. `PARTIAL`, `BLOCKED`, `FAIL`, `NOT PASSED`, source-only fixes, deferred workflow checks, and conditional evidence stay unchecked with notes.
